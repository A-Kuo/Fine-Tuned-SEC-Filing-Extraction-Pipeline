## Engineering Insights
- Production financial-extraction write-ups (Alan eng blog, Databricks) consistently frame **Pydantic/schema validation with a human-review fallback as table stakes, not a nice-to-have** — this repo already has the design language for it (dataclass `ExtractionResult`, JSON-schema in `schemas/pipeline-v1.json`) but the actual Pydantic validator + review-queue code lives only in `AgentOps/staged/`, unmerged.
- **Hybrid LLM + deterministic-rules architecture** is the recommended production pattern: use the LLM for semantic understanding (which section says what) and deterministic logic/regex for the parts where precision is non-negotiable (currency parsing, number-format normalization) — this repo's 5-strategy JSON cascade already follows this philosophy for parsing; the same pattern could extend to numeric normalization (`$394.3 billion` vs `394328000000`).
- **Multi-agent verifier patterns** (2026 production benchmarking paper) show that a dedicated verifier stage doing cross-field consistency checks (e.g., does `net_income` plausibly relate to `revenue` and `total_assets` for this sector?) catches errors single-pass extraction misses — a lightweight version of this maps directly onto the amendment-chain/temporal check recommended below.
- **Chunking for SEC filings works best when section-aware, not fixed-size** — parent-child chunking (small chunk for match precision, larger parent chunk returned for LLM context) is specifically called out as effective for tables/XBRL blocks; worth a quick audit of whether this repo's ingestion path (`scripts/parse_xbrl.py`, `scripts/fetch_edgar.py`) already does section-aware splitting or relies on fixed windows.
- **Fin-RATE's core finding — LLMs are good at single-document extraction but unreliable at tracking the same entity over time or comparing across entities — is a direct, testable claim against this repo's own architecture**, since the pipeline currently extracts each filing independently with no persisted "prior filing for this ticker" lookup before returning a result.

## Recommended Next Actions

Add a temporal/amendment-chain consistency validator (new module) that, for a given ticker + fiscal period, checks the newly extracted filing against the most recent prior extraction for the same ticker in PostgreSQL: flag (a) implausible metric jumps (e.g., >50% single-quarter revenue swing with no accompanying qualitative signal) and (b) supersession — a 10-K/A or 10-Q/A for the same ticker+period should mark the prior non-amended extraction stale rather than letting both stand as independent rows.
