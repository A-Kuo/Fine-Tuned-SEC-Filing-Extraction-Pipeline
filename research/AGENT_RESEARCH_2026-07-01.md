## Summary
2025–2026 work on SEC filing extraction has converged on three themes directly relevant to this repo: (1) PEFT/LoRA fine-tuning benchmarked specifically against XBRL-tagged SEC data rather than generic financial text, (2) document-structure-aware chunking (parent-child, section-aware) replacing naive fixed-size chunking for tables and XBRL tags, and (3) Pydantic-schema validation with human-review fallback as the standard production pattern for LLM extraction pipelines. AI-driven semantic extraction is now reported to catch materially more risk disclosures than keyword screening, reinforcing the value proposition of this repo's approach.

## Engineering Insights
- **Section/structure-aware chunking beats fixed-size chunking for filings.** Parent-child chunking (small child chunk for precise retrieval match, larger parent chunk returned for context) is reported to work well for tables and figures inside 10-Ks — worth testing against this repo's current chunking strategy if one exists. ([Snowflake eng blog](https://www.snowflake.com/en/engineering-blog/impact-retrieval-chunking-finance-rag/), [Databricks](https://community.databricks.com/t5/technical-blog/the-ultimate-guide-to-chunking-strategies-for-rag-applications/ba-p/113089))
- **Prepend a document-level summary to every chunk.** Prepending company name, filing date, form type, and key metrics to each chunk before embedding is reported to measurably improve retrieval precision on financial documents. Cheap to implement, high leverage.
- **Validate LLM output against a strict schema (Pydantic or equivalent) with a human-review fallback path** rather than trusting raw generation — this is now described as the standard production pattern for LLM extraction pipelines, not an edge case. ([Alan eng blog](https://medium.com/alan/lessons-from-running-an-llm-document-processing-pipeline-in-production-33d87f99cdb1))
- **Route documents by type before extraction.** Splitting multi-section filings and routing each section (MD&A, risk factors, financial statements) to a specialized extraction path reduces token usage and error rates versus one-shot whole-document extraction.
- **Consider few-shot/in-context alternatives alongside fine-tuning.** Several production write-ups now argue that 3 well-chosen in-context examples resolve many edge cases that would otherwise require a retraining cycle — worth having as a fallback lane for filing types this repo's fine-tuned model handles poorly.

## Recommended Next Actions
1. Add exact-tag/exact-span match metrics (not just token-level F1) to the eval suite, following the gap identified in the FinTagging benchmark — SEC/XBRL extraction is judged on precise tag and value matches, and this repo's evaluation currently doesn't appear to test for that specificity.
2. Implement Pydantic (or equivalent) schema validation on model output with a human-review queue for failed validations, matching the production pattern documented in the Alan/ZenML write-ups — this closes the gap between "demo extraction" and "production-grade pipeline."
3. Prototype parent-child section-aware chunking for tables/XBRL blocks and A/B it against the current chunking approach on a small filing sample, citing the Snowflake finance-RAG chunking study as the reference methodology.

## Security Notes
Public financial-LLM repos in this space commonly commit sample filing data or cached API responses that include SEC EDGAR user-agent strings or third-party data-vendor API keys (e.g., Calcbench/Intrinio) in notebook outputs. No evidence this repo has this issue, but worth a targeted grep of notebook cell outputs and `.env`-style files before the next public push, since this repo pulls from EDGAR and possibly other financial data APIs.
